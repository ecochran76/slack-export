## Slack mirror / daily briefing hardening

- Investigated why the daily briefing sometimes returned 0 Slack results.
- Root cause found: `slack-mirror` runtime was cwd-sensitive because the repo-local config used a relative DB path (`./.local/state/slack_mirror_test.db`), and live/systemd flows were still pointing at `config.local.yaml`. The daily briefing was also using a previous-calendar-day window and `search keyword`, which is ranked/capped rather than exhaustive.
- Repo fixes implemented and committed:
  - `045cd73` — `Harden slack-mirror config and storage resolution`
    - config discovery order: `./config.local.yaml`, `./config.yaml`, `~/.config/slack-mirror/config.yaml`
    - resolve `dotenv`, `storage.db_path`, and `storage.cache_root` relative to the config file instead of cwd
    - user-scope defaults for DB/cache
    - `scripts/user_env.sh` migration to stable user-scope state/cache
    - added raw `slack-mirror messages list`
    - docs/tests updated
  - `0809c71` — `Default live mode to user-scope config`
    - live-mode systemd/tmux helpers now prefer `~/.config/slack-mirror/config.yaml`
    - live-mode docs updated
- Migrated slack-mirror runtime to user scope:
  - config: `~/.config/slack-mirror/config.yaml`
  - DB: `~/.local/state/slack-mirror/slack_mirror.db`
  - cache: `~/.local/cache/slack-mirror`
  - ran `scripts/user_env.sh install`
  - verified both workspaces (`default`, `soylei`) authenticate successfully
  - one-time sync populated the user DB (`default`: 621 channels / 262 messages in the sync pass; `soylei`: 158 channels / 43 messages)
- Patched local daily briefing script at `~/.openclaw/workspace/scripts/daily_briefing.py` to:
  - use user-scope config by default
  - query Slack with a rolling last-24h window
  - use `slack-mirror messages list` instead of `search keyword`
  - fail loudly when mirror data is stale instead of silently treating stale data as zero activity
- Remaining unresolved issue:
  - live Slack dynamic updates into the user DB are still not confirmed healthy
  - services are active and now point at the user config, but the latest message in the user DB stayed at ts `1773192878.283149` (`"OK go"`, around 20:34 CDT)
  - later thread messages (including the 20:48 and 21:03 checks) were missing from the user DB
  - next debugging target is the Socket Mode / event ingestion path, not config or DB location
